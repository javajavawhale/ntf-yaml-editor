# NTF YAML セル差分設計

## 目的

NTF YAML セル差分は、YAML 化されたテストデータを構造化してレビューするための機能である。Excel から YAML へ移行した後も、表としての概念を保ち、ローカル編集やレビューでテストデータ破壊を見つけやすくする。

目的は汎用的なテキスト差分を見せることではない。NTF YAML を表として解釈し、テストデータの概念モデルに沿って、変更されたセル、行、ブロック、シート、ファイルを強調表示する。

主な利用者:

- AI または人間が編集したテストデータをローカルで確認するレビューイ。
- 前回レビューしたリビジョンと今回レビューするリビジョンの差分を確認するレビュワー。

## スコープ

対象:

- VS Code 拡張のセル差分ビュー。
- ローカル CLI による単一 HTML レポート生成。
- VS Code と HTML で共有する差分モデルと単一表統合ビュー。
- 2 つの Git ref 間で変更された YAML ファイル。

対象外:

- CI 成果物生成。
- CI 設定サンプル提供。
- PR コメント投稿。
- HTML レポート CLI での作業ツリー比較。
- NTF ではない汎用 YAML の差分表示。

## 主なワークフロー

### VS Code

拡張は、Git 関連のローカルレビュー文脈から `Open NTF YAML Cell Diff` を開ける導線を提供する。

理想は、Git の変更一覧や差分エディタ周辺からセル差分ビューを開くことである。VS Code API の制約で組み込みの Git 差分エディタを置き換えられない場合は、同じセル差分ビューをコマンド、コンテキストメニュー、エディタタイトルアクションから開く。

代替導線で変わるのは開き方だけである。差分の見え方や意味論は変えない。

### HTML レポート

CLI は、明示された 2 つの Git ref からローカルの単一 HTML レポートを生成する。

```sh
ntf-yaml diff --base <git-ref> --head <git-ref> [-o ntf-yaml-diff.html]
```

ルール:

- `--base` と `--head` は必須。
- 値にはコミット ID、ブランチ、タグ、`HEAD~n` など、任意の Git ref を指定できる。
- 比較対象は Git ref 同士のみ。作業ツリーは比較しない。
- `-o` を省略した場合は `ntf-yaml-diff.html` に出力する。
- YAML 差分がない場合も HTML を生成する。
- レポート生成に成功した場合は終了コード `0`。
- Git エラー、解析エラー、書き込みエラーは非 `0`。

## ファイル選択

レポート対象は、`--base` と `--head` の間で変更されたすべての `.yaml` / `.yml` ファイルである。

将来的には、プロジェクト規約を `.ntf.yaml` に寄せ、既定の対象もそれに合わせて狭める。

ルール:

- Git のリネーム検出を使う。
- リネーム / 移動は、旧パスと新パスを持つ同一ファイルとして扱う。
- 追加された YAML ファイルは、全セル追加として表示する。
- 削除された YAML ファイルは、全セル削除として表示する。
- 対象 YAML の base/head いずれかが NTF YAML として解析できない場合、コマンドを失敗させ、誤解を招くレポートは生成しない。

## 表示モデル

VS Code と HTML は、同じ表示モデルとして単一表統合ビューを使う。

主要設計として左右 2 ペインの差分表示は使わない。NTF YAML の表は横長になりやすく、左右分割では変更セルを追いにくくなる。

表示ルール:

- 変更後 YAML の順序を、行・列の主たる表示順にする。
- 削除された行や列は、旧位置に近い場所へ挿入する。
- 変更セル内に変更前 / 変更後の値を上下に表示する。
- 変更セルを強くハイライトする。
- 変更行を薄くハイライトする。
- 追加行と削除行を行単位で区別する。
- 初期表示は変更があるシート / ブロックを中心にする。
- 変更があるブロックでは、変更行の全セルを表示する。
- 未変更行は初期状態では隠す。ただし文脈行は表示する。
- 追加行・削除行の前後 1 行に未変更行があれば文脈として表示する。
- ブロック全体の全行を展開できる導線を持つ。

## 表差分の意味論

差分エンジンは、NTF YAML を生のテキスト行ではなく表の概念で比較する。

### ListMap / 表ブロック

対象:

- `LIST_MAP`
- `SETUP_TABLE`
- `EXPECTED_TABLE`

ルール:

- 行の同一性は、利用可能なら `no` または `"[no]"` を優先する。
- 列の同一性は列名で判断する。
- 行の同一性に使う値が変わった場合は、旧行削除 + 新行追加として扱う。

### RawRows ブロック

対象:

- `SETUP_VARIABLE`
- `EXPECTED_VARIABLE`

ルール:

- 行の同一性は行番号で判断する。
- 列の同一性は列番号で判断する。
- セルは行番号と列番号で比較する。

### 固定長ファイルブロック

対象:

- `SETUP_FIXED`
- `EXPECTED_FIXED`

設計は Nablarch Testing Framework の固定長ファイル概念に従う。固定長ファイルブロックは、レコード種別ごとの表として表示する。

ルール:

- ブロックをレコード種別ごとに分割する。
- 各レコード種別を別表として表示する。
- 行の同一性は、レコード種別内の行番号で判断する。
- 列の同一性はフィールド名称で判断する。
- データ型とフィールド長は列ヘッダ属性として扱う。
- データ型またはフィールド長が変わった場合は、列ヘッダをハイライトする。

## レポート内容

HTML レポートには以下を含める。

- base ref
- base コミット SHA
- head ref
- head コミット SHA
- 生成日時
- リポジトリパス
- 変更ファイル数
- 追加ファイル数
- 削除ファイル数
- 変更シート数
- 追加シート数
- 削除シート数
- 変更ブロック数
- 追加ブロック数
- 削除ブロック数
- 変更行数
- 追加行数
- 削除行数
- 変更セル数
- 追加セル数
- 削除セル数

レポートは、インライン CSS、インライン JavaScript、埋め込み差分データを持つ単一 HTML ファイルとする。外部 CDN には依存しない。

## 実装方針

主要な実装単位は、VS Code Webview と HTML レポートの両方で描画できる構造化差分モデルである。

推奨レイヤ:

1. NTF YAML を既存エディタモデルへ解析する。固定長ファイル表にも拡張する。
2. ファイル、シート、ブロック、表、行、列、セルの比較用同一性へ正規化する。
3. 構造化差分モデルを生成する。
4. 差分モデルを共有の単一表統合ビューとして描画する。
5. VS Code にローカルセル差分ビューの入口を追加する。
6. ローカル HTML レポート生成用に `ntf-yaml diff --base --head` を追加する。
