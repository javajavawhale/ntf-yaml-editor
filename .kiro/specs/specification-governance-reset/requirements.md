# 要件: 仕様統制の立て直し

## 目的

この spec は、実装が暗黙の思い込みで進まないように、NTF YAML Editor の開発手順を「仕様を明示してから実装する」形へ戻すためのものです。

根本課題は、NTF YAML の解釈、ビューごとの責務、要確認の設計判断が、レビュー可能な仕様に紐づかないままコードへ固定化されていることです。

## スコープ

対象:

- NTF YAML Editor の変更を仕様起点で進めるための手順を定める。
- 確定事項、要確認事項、却下事項を分離する。
- Normal Editor、SCM Diff、Cell Diff、HTML Report のビュー責務を明確にする。
- 実装タスクがどの仕様判断に基づくかを追跡できるようにする。
- テストを「偶然の現行挙動」ではなく「レビュー済み仕様」を守るものにする。

対象外:

- この spec で parser、serializer、diff、Webview を直接リライトすること。
- NTF YAML の全ブロック仕様をこの場ですべて決め切ること。
- 現行実装を丸ごと置き換えること。

## 要件

### R1: 仕様の正本

非自明な実装変更の前に、NTF YAML の振る舞いを説明するレビュー済み仕様を用意すること。

受け入れ条件:

- 振る舞いを変える実装タスクは、該当する spec の節を参照している。
- 確定事項、要確認事項、却下事項が見て分かる形で分離されている。
- コードコメントを振る舞い仕様の主な正本にしない。

### R2: ビュー責務と共有仕様の明示

次の4種類のビューについて、共有する振る舞いとビュー固有の責務を記述すること。

- Normal Editor
- SCM Diff
- Cell Diff
- HTML Report

受け入れ条件:

- 仕様に、ビュー間で共有する振る舞いと、ビューごとに異なる振る舞いが書かれている。
- SCM Diff と Cell Diff を同じものとして扱っていない。
- 表示構造、セルの意味、diff の同一性ルール、構造セルの扱い、readonly の基本方針は、可能な限り共有する。
- 分割ペイン、差分操作、エクスポート専用 UI など、ビュー固有の責務がどの層にあるかを明示している。

### R3: NTF YAML 解釈の統制

NTF YAML の解釈は、実装の副作用ではなく設計判断として扱うこと。

受け入れ条件:

- 対応する各ブロック種別について、parse、render、edit、save、diff の振る舞いが記述されている。
- ファイル系ブロックについて、表示用の構造化と保存用の構造化を区別している。
- 確定事項と要確認事項が、ラベルなしで同じ節に混在していない。

### R4: NTF 実行可能性の優先

このエディタが扱う YAML は、Nablarch Testing Framework で実行可能なテストデータであることを前提にすること。

受け入れ条件:

- 実装判断では、汎用 YAML として自然かどうかより、NTF テストデータとして有効かどうかを優先する。
- エディタの表示、編集、保存、変換、差分表示によって、NTF で実行可能だった YAML を実行不能にしてはならない。
- NTF 仕様に反する表示、編集、保存仕様は、便利に見えても採用しない。
- NTF 仕様が未確認の振る舞いは、確定仕様として扱わない。

### R5: example 由来 YAML の実証基盤化

仕様判断では、Nablarch example 由来の実行可能な NTF YAML を主要な実証基盤として使うこと。

受け入れ条件:

- 未確定の YAML 形状は、まず example 由来の実例で確認する。
- example に存在する形状と、推測または合成 fixture で作った形状を区別する。
- example に存在しない形状をサポートする場合は、NTF 仕様または明示的な設計判断を根拠として記録する。
- example 由来 YAML に対する表示、編集、保存、差分の仕様は、代表 fixture または実例への参照で検証できる。

### R6: 要確認事項の明示

根拠不足、文書矛盾、実行可能性未確認、またはユーザー判断が必要な振る舞いは、要確認事項として明示すること。

受け入れ条件:

- 要確認事項には、何を確認する必要があるかが書かれている。
- 一時対応を入れる場合は、理由、見直し条件、撤去条件が書かれている。
- 要確認事項を守るテストは、最終仕様のテストに見えない名前または説明になっている。

### R7: テストと仕様の対応

テストは、レビュー済み仕様を守るために使うこと。偶然の現行挙動を無条件に守らないこと。

受け入れ条件:

- 矛盾している仕様や未レビューの仮定を固定している既存テストは、見直し対象として扱われる。
- 新しい characterization test は、確定仕様と現行挙動の記録を区別している。
- ドメイン依存の振る舞いは、代表的な NTF YAML fixture を優先して検証する。

### R8: リファクタリングのゲート

リファクタリングは、明示された責務境界に基づいて行うこと。

受け入れ条件:

- リファクタリングタスクは、改善する責務境界を明記している。
- 責務境界は、既存ファイル名ではなく、入力取得、NTF YAML 解釈、NTF 検査、編集操作、保存、差分、表示、外部入口の処理の流れから説明されている。
- UI 都合のロジックを NTF YAML 解釈や保存の責務へ移す場合は、それが NTF 仕様または example 由来 YAML に基づく判断であることを spec が説明している。

### R9: リライト判断のゲート

全面的なリライト判断は、仕様整理と characterization によって具体的な失敗モードが見えてから行うこと。

受け入れ条件:

- リライト提案は、どの層を置き換えるべきかを明示している。
- リライト提案は、失敗した要件または制御不能な結合を根拠としている。
- fixture、手動テスト計画、docs、VS Code integration などの資産は、明示的に無効と判断されない限り残す。
