# 設計: 仕様統制の立て直し

## 課題の捉え方

このプロジェクトは、実装が共有された仕様理解より先に進んだことで保守しづらくなっています。

起きている失敗パターンは次の通りです。

1. NTF YAML の振る舞いとして機能が要求される。
2. 実装時に、不足している仕様をその場の推測で補う。
3. その推測に基づいてテストが追加される。
4. 後から、その推測が誤りだったことが分かる。
5. 誤った前提が parser、serializer、diff、Webview、fixture、docs に広がっている。

この立て直しでは、NTF YAML の解釈、ビュー挙動、保存結果に影響するコードを書く前に、仕様解釈をレビュー可能な状態に戻します。

## この spec 固有の運用ルール

NTF YAML Editor では、NTF YAML の意味や、NTF テストデータとしての正しさを変える可能性がある変更について、実装前に判断根拠を記録します。

判断根拠を記録する代表的な変更は次の通りです。

- ブロック種別の解釈。
- 編集可否。
- 保存後 YAML。
- diff の同一性ルール。
- ビューごとの表示・操作差。
- example にない YAML 形状の扱い。

判断基準は、変更が NTF テストデータの意味、実行可能性、保存結果、レビュー結果に影響するかどうかです。

NTF YAML の意味、実行可能性、保存結果、レビュー結果を変えない小さなバグ修正や UI 調整は、フル spec を省略してよいものとします。

## 文書の使い分け

このプロジェクトでは、文書を次のように使い分けます。

- `.kiro/steering/`: 長期的に安定させるプロジェクト原則を書く。
- `.kiro/specs/`: 特定の変更領域について、レビュー済みの判断根拠と実装計画を書く。
- `docs/`: 将来の保守者やレビュワーが実装計画なしで読むべき、長期的な仕様や検証文脈を書く。

Spec で仕様を確定した場合、将来の保守者が実装計画なしでも読むべき内容は `docs/` に反映します。長期的な原則が変わる場合だけ `.kiro/steering/` を更新します。

## 仕様状態

NTF YAML の意味、実行可能性、保存結果、レビュー結果に影響する振る舞いは、次のいずれかに分類します。

- 確定: 判断根拠と影響範囲が記録済みで、実装と仕様テストへ進めてよい。
- 要確認: 根拠不足、文書矛盾、実行可能性未確認、またはユーザー判断が必要な状態。最終仕様として実装しない。
- 却下: 採用しない判断が記録済みで、再提案時には却下理由を確認する。

## 判断根拠の優先順位

NTF YAML の解釈や保存結果を決めるときは、次の順に根拠を優先します。

1. Nablarch Testing Framework の仕様。
2. Nablarch example 由来の、実行可能な NTF YAML。
3. 既存 converter output。
4. 仕様確認のために作った合成 fixture。
5. 既存実装の都合。

既存実装の都合は、上位の根拠と矛盾する場合は採用しません。

根拠を使うときは、docs URL、知識ファイル、example path、converter output path、調査メモ、または spec の該当節への参照を残します。「NTF 仕様ではこう」と書く場合も、記憶や推測だけを根拠にしません。

example に存在する形状と、合成 fixture や推測で作った形状は区別します。example に存在しない形状をサポートする場合は、NTF 仕様または明示的な設計判断を根拠として記録します。

既存 converter output は、example が不足している場合の補助根拠として使えます。ただし converter output だけで確定仕様にする場合は、その YAML が NTF テストデータとして実行可能であること、または実行可能性を確認する計画があることを記録します。

## ユーザー確認が必要なタイミング

次の状況では、実装前にユーザー確認を行います。

- NTF 仕様または example 由来 YAML で確認できない形状を、対応対象に含めるとき。
- converter output または合成 fixture を主な根拠にして、保存、編集、diff の仕様を決めるとき。
- `docs/`、`.kiro/specs/`、`.kiro/steering/`、既存テストの間で仕様矛盾があり、どれを採用するか判断が必要なとき。
- 編集可否、保存後 YAML、diff の同一性ルール、ビューごとの表示・操作差を変更するとき。
- 確定済み仕様を変更、撤回、または `要確認` に戻すとき。
- NTF 実行可能性を確認できないまま、一時対応を入れる必要があるとき。

ユーザー確認では、判断対象、根拠、影響するビュー、保存影響、差分影響、残るリスクを示します。確認結果は spec の該当節または task に記録します。

## 文書矛盾の解消

`docs/`、`.kiro/specs/`、`.kiro/steering/` の間で矛盾を見つけた場合、片方を暗黙に正として扱いません。

解消手順:

1. 矛盾している文書と節を spec に記録する。
2. 影響するビュー、保存、差分、テストを確認する。
3. 判断根拠の優先順位に従って、確定、要確認、却下のいずれかに分類する。
4. 確定した内容を `docs/` に反映する。要確認の内容は spec に判断待ちとして残す。
5. 長期的な原則が変わる場合だけ `.kiro/steering/` を更新する。

既存 docs の両論併記は、仕様レビュー中の一時状態としては許容します。ただし、実装タスクへ進む前に、該当箇所を確定または要確認としてラベル付けします。

## ビュー分類と共有方針

UI または diff に関わる spec は、対象ビューと共有する振る舞いを記述します。

表示構造、セルの意味、diff の同一性ルール、構造セルの扱い、readonly の基本方針は、可能な限りビュー間で共有します。仕様変更は共有コンポーネントと NTF YAML の構造解釈へ反映し、Normal Editor、SCM Diff、Cell Diff、HTML Report へ伝播させます。

ビューごとの違いは、ビューの所有者や操作文脈から必要になるものだけを明示します。

| ビュー | 所有者 | 主な責務 |
| --- | --- | --- |
| Normal Editor | extension の custom text editor + Webview | 単一ペインでの表編集と保存。 |
| SCM Diff | VS Code diff editor + 独立した2つの custom editor pane | VS Code が所有する分割ペイン内で base/head をそれぞれ描画する。 |
| Cell Diff | extension が所有する Webview panel | レビュー用ワークフロー、分割/統合表示、ref 選択、export 導線。 |
| HTML Report | 生成された standalone HTML | VS Code 操作なしの静的レビュー出力。 |

ビューごとの差分が必要な場合は、その理由と影響を spec に記録します。たとえば SCM Diff は VS Code が分割ペインを所有し、Cell Diff と HTML Report は extension 側の diff 表示を使うため、操作 UI の責務が異なります。

## 責務境界の洗い出し方

責務境界は、既存ファイル名や実装都合ではなく、NTF YAML Editor が扱う処理の流れから洗い出します。

洗い出し対象の流れ:

1. 入力を得る: VS Code document、Git ref、working tree、index、converter output、example YAML。
2. NTF YAML として解釈する: sheet、block、row、cell、file block などの意味を決める。
3. NTF テストデータとして検査する: 実行可能性、参照関係、構造の不整合を確認する。
4. ユーザー操作を受ける: 表示、編集、並び替え、readonly、保存操作を扱う。
5. YAML として保存する: spec 上安全な変更だけを YAML へ戻す。
6. 変更を比較する: sheet、block、row、cell の同一性ルールで差分を作る。
7. ビューへ表示する: Normal Editor、SCM Diff、Cell Diff、HTML Report で共有表示とビュー固有 UI を組み立てる。
8. 外部入口へ接続する: VS Code command、custom editor、CLI command、Git context とつなぐ。

この流れから、次の責務候補をレビューします。

| 責務候補 | 判断すること |
| --- | --- |
| 入力取得 | どの入口から base/head/current YAML を取得するか。 |
| NTF YAML 解釈 | YAML 形状をどの NTF 概念として扱うか。 |
| NTF 検査 | 実行可能性や構造不整合をどこまで検出するか。 |
| 編集操作 | どの操作を許可し、どの操作を readonly または要確認にするか。 |
| 保存 | どの変更を YAML へ戻してよいか。 |
| 差分 | 何を同一 sheet/block/row/cell とみなすか。 |
| 表示 | どの表示仕様をビュー間で共有し、どこをビュー固有にするか。 |
| 外部入口 | VS Code と CLI の入口がどの共有責務を呼ぶか。 |

レビュー観点:

- NTF 仕様または example 由来 YAML に基づく判断が、表示や UI 都合の責務に埋もれていないか。
- 保存してよい変更と、表示・差分のためだけに構造化する情報が混ざっていないか。
- ビュー間で共有すべき仕様が重複実装になっていないか。
- SCM Diff、Cell Diff、HTML Report など入口固有の制約が、NTF YAML の意味論に漏れていないか。

実装時のファイル分割は、この責務レビューの後に決めます。既存の `Parser`、`Model`、`Serializer`、`Diagnostics`、`Diff`、`Webview`、`Extension`、`CLI` という名前は候補であり、責務レビューの結果に合わせて統合、分割、改名できます。

## テスト戦略

テストは意図ごとに分類します。

- 仕様テスト: NTF 仕様または example 由来 YAML に紐づく確定仕様を守る。
- Characterization test: 危険なリファクタリングの前に、現行挙動を記録する。
- Regression test: 修正済みバグの再発を防ぐ。
- 要確認テスト: 一時対応や現行挙動を守るが、確定仕様として扱わない。

要確認テストは、最終的なプロダクト仕様に見えない名前またはコメントを持たせます。

仕様テストには、根拠にした NTF 仕様、example path、converter output path、または spec 節を分かる形で残します。合成 fixture だけを根拠に仕様テストを作る場合は、なぜ example 由来 YAML で確認できないかを記録します。

## テスト配置

テストコードとテストデータは、仕様との対応を追える場所に置きます。

配置方針:

- 自動テストコードは `test/` に置き、テスト名またはコメントから根拠にした spec 節、example path、fixture が分かるようにする。
- example 由来 YAML は、代表 fixture として `test/fixtures/ntf-samples/` または用途別 fixture に置く。
- 手動確認用 fixture は `test/fixtures/manual/` に置き、自動テストの正本にしない。
- diff シナリオ用 fixture は `test/fixtures/diff-scenarios/` に置く。
- 合成 fixture は、example 由来 YAML と混ざらない名前またはディレクトリにする。
- 生成 HTML、スクリーンショット、パッケージ成果物などの出力物は、仕様根拠として扱わない。

fixture を追加するときは、example 由来、converter output 由来、合成、手動確認用のどれかが分かるようにします。

## リライト戦略

初期判断として、全面リライトは行いません。

この立て直しで、置き換えるべき最小の層を見極めます。

- raw preservation と NTF YAML 解釈を明示できない場合のみ、parser / serializer の置き換えを検討する。
- 文書化された同一性ルールを現在の diff 構造で表せない場合のみ、diff の置き換えを検討する。
- model と diff の責務が安定してから、Webview 内部の分割または置き換えを検討する。
- fixture、手動テスト計画、動作している integration 経路は、却下された仕様を固定している場合を除いて残す。

## 直近の仕様リスク

次の領域は、これ以上実装を進める前にレビューが必要です。

- 固定長ファイルブロックの意味論。
- ファイル系ブロックにおける表示用構造と保存用構造の違い。
- ファイル系ブロックのどの操作を readonly、要確認、確定仕様とするか。
- SCM Diff と Cell Diff の責務差。
- 既存 glossary / steering の canonical form 表現が、合意済みの根本課題より強くなっていないか。
- 既存テストのうち、偶然の挙動を固定しているものがどれか。
