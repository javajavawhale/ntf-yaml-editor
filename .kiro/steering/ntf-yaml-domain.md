# NTF YAML ドメイン Steering

## ドメインモデル

このプロジェクトは NTF YAML エディタおよび CLI ツールチェインであり、汎用 YAML エディタではない。

プロダクトは、Nablarch Testing Framework の Excel テストデータが持っていた表の概念モデルを、YAML 移行後の編集、診断、レビュー、差分でも保つ。

- トップレベル YAML キーはシート。
- シート配下のブロックはテストデータセクション。
- 重要なブロックの多くは表。
- 編集とレビューはセル、行、列、ブロック、シート、ファイルの単位で語る。

## 対応する表の種類

キー付き表ブロック:

- `LIST_MAP`
- `SETUP_TABLE`
- `EXPECTED_TABLE`

行番号付きブロック:

- `SETUP_VARIABLE`
- `EXPECTED_VARIABLE`

固定長ファイルブロック:

- `SETUP_FIXED`
- `EXPECTED_FIXED`

固定長ファイルブロックは、レコード種別ごとの表としてモデル化する。各レコード種別内ではフィールド名称を列の同一性とし、データ型とフィールド長は列ヘッダ属性として扱う。

## シリアライズ前提

NTF YAML は正規化してシリアライズされた YAML として運用する。

セル差分モデルでは、書式だけの変更とテストデータ変更を区別する必要はない。正規化されていない YAML がワークフローに入った場合は、構造化レビューを期待する前にプロジェクトのシリアライザで正規化する。

シリアライズは、移行後の YAML テストデータを安定してレビューするための前提である。人間や AI が編集した結果も、同じモデルを通して保存し、意図しない破壊を検出しやすくする。

## 差分原則

セル差分は構造化レビューであり、テキスト差分ではない。

- 類似度による対応付けよりも、決定的な同一性ルールを優先する。
- 行の同一性が変わった場合は、旧行削除 + 新行追加として扱う。
- キー付き表では、利用可能なら `no` または `"[no]"` を優先的な行の同一性とする。
- RawRows では行番号と列番号を使う。
- 固定長ファイルのレコード表では行番号とフィールド名称を使う。
- 表示順は変更後の順序を主とし、削除された要素は旧位置に近い場所へ挿入する。

## 視覚レビュー原則

VS Code と HTML レポートは、同じ単一表統合ビューを共有する。

- 変更セルは同じセル内に変更前 / 変更後の値を表示する。
- 変更行は薄く行ハイライトする。
- 追加行と削除行は視覚的に区別する。
- 初期表示は変更があるシートとブロックを中心にする。
- 必要に応じて表全体の文脈へ展開できる。

プロダクトの主要なセル差分表現として左右分割差分は避ける。NTF YAML の表は横長になりやすく、単一表統合ビューの方が表の概念を保てる。
