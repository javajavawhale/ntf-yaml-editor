# NTF YAML ドメイン Steering

## ドメインモデル

このプロジェクトは NTF YAML エディタおよび CLI ツールチェインであり、汎用 YAML エディタではない。

プロダクトは、Nablarch Testing Framework の Excel テストデータが持っていた表の概念モデルを、YAML 移行後の編集、診断、レビュー、差分でも保つ。

- トップレベル YAML キーはシート。
- シート配下のブロックはテストデータセクション。
- 重要なブロックの多くは表。
- 編集とレビューはセル、行、列、ブロック、シート、ファイルの単位で語る。

## 対応する表の種類

通常 table:

- ファイル指定を持つ `VARIABLE` / `FIXED` 系以外の NTF block

ファイル系 table:

- `SETUP_VARIABLE`
- `EXPECTED_VARIABLE`
- `SETUP_FIXED`
- `EXPECTED_FIXED`

固定長ファイルブロックは、可変長ファイルブロックと同じファイル系表として扱う。ディレクティブ行は固定長ファイル固有の値を持つが、ディレクティブ行以外の表構造は可変長ファイルブロックと同じである。

## シリアライズ前提

NTF YAML は、表示・編集・diff の意味を保ったまま保存できることを優先する。

canonical form は、converter output、editor save、CLI format の表記を将来的に揃えるための長期目標である。
ただし、現時点で保存処理を独立した Serializer 層として設計しない。

保存結果は NTF 実行可能な YAML として扱えることを優先し、エディタ都合の表記変換を仕様として先行させない。

## 差分原則

セル差分は構造化レビューであり、テキスト差分ではない。

- 類似度による対応付けよりも、決定的な同一性ルールを優先する。
- 行の同一性が変わった場合は、旧行削除 + 新行追加として扱う。
- 通常 table では、利用可能なら `no` または `"[no]"` を優先的な行の同一性とする。
- ファイル系 table では行番号と列番号を使う。
- 表示順は変更後の順序を主とし、削除された要素は旧位置に近い場所へ挿入する。

## 視覚レビュー原則

SCM diff、Cell Diff、HTML レポートは、同じ差分意味論を共有する。

- Cell Diff と HTML レポートの unified 表示では、変更セルは同じセル内に変更前 / 変更後の値を表示する。
- 変更行は薄く行ハイライトする。
- 追加行と削除行は視覚的に区別する。
- 初期表示は変更があるシートとブロックを中心にする。
- 必要に応じて表全体の文脈へ展開できる。

SCM diff は VS Code が左右ペインを管理する。
Cell Diff と HTML レポートは split / unified 表示を提供する。
